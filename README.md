# unifying_simulation — AOG 자재 수급 어시스턴트

AOG(Aircraft on Ground) 상황 발생 시, 정해진 절차에 따라 자재를 수급하고 각 단계에서
실제로 연락/발송/조회까지 이어갈 수 있도록 돕는 Streamlit 대시보드입니다.
5분 내 전 프로세스를 끝내야 하는 실제 AOG 상황을 전제로, **속도와 정해진 프로세스의 정확성**을
최우선 설계 원칙으로 삼았습니다. Google Colab의 노트북 하나(`AOG_Dashboard.ipynb`)로 배포됩니다.

## 실행 방법

1. `AOG_Dashboard.ipynb`를 [Google Colab](https://colab.research.google.com/)에서 엽니다.
2. **Cell 1 → Cell 2 → Cell 3 순서대로** 실행합니다.
   - Cell 1은 이제 런타임을 강제로 재시작하지 않습니다(아래 "이번 업데이트" 참고). 설치가 끝나면 바로 Cell 2로 넘어가세요.
   - Cell 3은 `google.colab.output`으로 앱을 새 창에 띄웁니다. npm/localtunnel 같은 제3자 터널을 전혀 쓰지 않으므로 사내망 보안 정책에 걸릴 일이 없습니다.
3. 새 창에서 대시보드가 열리면 사이드바에서 **"🚨 AOG 대시보드" / "🗺️ 실시간 운항 현황" / "⚙️ 데이터 관리"** 중 화면을 선택합니다.

## 이번 업데이트 내용 (v3)

### 1. AI 속도·역할 재조정 — "AI는 개입이 아니라 추천만"
- **문제**: 이전 버전은 케이스 상태가 바뀔 때마다 로컬 LLM이 자동으로 상황을 다시 생성해, 매 단계 클릭마다 몇 초씩 지연이 생겼습니다. 5분 내 전 프로세스를 끝내야 하는 도구로는 치명적인 결함이었습니다.
- **수정**: 좌측 "AI 상황 요약 & 추천 행동" 패널은 이제 **항상 규칙 기반으로 즉시(0초)** 표시됩니다. LLM은 이 패널에서 완전히 빠졌습니다.
- **AI의 역할은 4·5단계 긴급 메일 문구를 다듬는 것으로 한정**했습니다. 메일은 항상 **정해진 템플릿으로 즉시 완성**되어 바로 발송 가능하며, "✨ AI로 다듬기" 버튼을 눌렀을 때만 선택적으로 로컬 LLM이 문장을 다듬습니다(실패해도 템플릿 문구가 그대로 유지되어 앱이 멈추지 않습니다). 생성 길이도 줄여 응답 속도를 높였습니다.

### 2. "AI 비서"다운 조작감 — 채팅 입력 + 자동 순차 확인 + 승인만 요청
- 화면 하단에 **채팅 입력창**이 추가되었습니다. "B777-300ER 기종 ICN에서 AOG 났어, 부품 BRK-B777-CARBON-01 당장 필요해"처럼 자연어로 말하면, 규칙 기반 경량 파서(정규식)가 기종/자재넘버/공항을 즉시 추출해 프로세스를 시작합니다. 일부만 인식되면 무엇을 못 찾았는지 알려주고 아래 "직접 입력" 폼에 인식된 값을 채워 넣습니다.
- **1~3단계(FAK/Allocation/Pooling)는 재고가 없으면 사람 개입 없이 자동으로 다음 단계까지 확인**합니다 — 순수 내부 재고 조회라 판단이 필요 없기 때문입니다. **4·5·6단계는 외부 연락(메일/전화)이나 물리적 이동이 필요한 실제 의사결정 지점이라 항상 멈춰서** "✅ 네, 진행해주세요" / "➡️ 아니요, 다음 대안 찾기" 승인만 받습니다.
- **왜 진짜 LLM 함수 호출(Function Calling)·ReAct 루프를 쓰지 않았는가**: 1~6단계의 순서는 이미 회사 정책으로 고정된 절차이지, LLM이 매번 "무엇을 조회할지" 판단해야 하는 문제가 아닙니다. 그 판단을 LLM에 맡기면 각 단계마다 추론 지연(수 초)과 실수 가능성(잘못된 순서로 호출·환각)이 생기는데, 정작 evaluate_step()의 재고 조회 자체는 파이썬 딕셔너리 조회라 이미 0초에 끝납니다. 그래서 "자동 순차 확인 → 사람은 승인만" 이라는 **경험(사용자 체감)**은 그대로 구현하되, 그 내부 동작은 느리고 오류 가능성이 있는 LLM 루프 대신 **결정론적 파이썬 함수를 그대로 순회**하도록 설계했습니다. 정해진 프로세스의 정확성과 5분 내 완료라는 두 요구를 동시에 만족시키는 선택입니다.

### 3. 실시간 운항 현황 화면 — 시인성 개선
- 상단에 **KPI 카드**(전체 항공편 / 비행 중 / 예정 / 도착)를 추가했습니다.
- 항공편 표는 **상태 우선순위(비행 중 → 예정 → 도착) + 도착 임박 순으로 정렬**되고, 진행률은 막대(Progress Column)로 한눈에 보이도록 바꿨습니다.
- 공항별 정박 현황도 KPI 카드(전체 스탠드 / 평균 혼잡도 / 혼잡 공항 수)와 혼잡도 막대 + 🟢/🟡/🔴 상태 라벨로 재구성했습니다.

### 4. 데이터 관리 기본 데이터 확충
FAK·Allocation 재고를 기종당 2건씩(총 12건), Pooling 파트너사를 5곳→7곳, 공항을 10곳→12곳(방콕·시드니 추가), 기종별 운영 항공사를 3곳→평균 4곳, Allocation 부서 연락처를 5곳→7곳으로 늘렸습니다.

### 5. Cell 1 pip install 재디버깅
- **문제**: 이전 버전의 `--upgrade` + 강제 재시작(`os.kill`) 조합이 Colab 환경에서 여전히 실패한다는 재현 보고를 받았습니다.
- **조치**: `--upgrade`와 강제 재시작을 모두 제거했습니다. 이미 만족되는 패키지(Colab에 내장된 numpy/pandas 등)는 손대지 않도록 순수 `pip install`(업그레이드 없음)만 수행합니다 — 그러면 애초에 재시작이 필요한 버전 충돌 자체가 생기지 않습니다. `-q`(quiet)도 제거해 문제가 생기면 출력이 그대로 보이도록 했습니다.
- 그래도 위젯마다 `TypeError`가 나면, Colab 메뉴에서 **런타임 → 세션 다시 시작** 후 Cell 1은 건너뛰고 Cell 2부터 실행하는 수동 대응 경로를 노트북에 안내해두었습니다.

- **⌨️ 직접 입력** (접이식): 채팅으로 일부만 인식됐거나 수동 입력을 선호할 때 사용하는 구조화된 폼입니다.
- 좌측 절반 — **🤖 AI 상황 요약 & 추천 행동**: 항상 즉시 표시되는 규칙 기반 요약. 지금까지 조회 결과와 다음 추천 행동을 보여줍니다.
- 우측 절반 — **📋 프로세스 진행**: 1~3단계는 자동으로 훑고, 4단계 이상 또는 재고가 확인된 지점에서 멈춰 **"✅ 네, 진행해주세요"** / **"➡️ 아니요, 다음 대안 찾기"** 승인을 기다립니다.

### 단계별 종료 시 실제로 연결되는 조치
| 종료 단계 | 제공되는 조치 |
|---|---|
| 1단계 FAK | 케이스 시작 시 입력한 **요청 정비사 연락처**로 바로 걸 수 있는 `tel:` 링크 |
| 2단계 Allocation | 해당 공항의 **Allocation 부서** 전화(`tel:`)·메일(`mailto:`) 링크 |
| 3단계 Pooling | 매칭된 **Pooling 파트너사** 전화·메일 링크 |
| 4단계 (Main Station 타사) | 재고가 확인되면 해당 항공사 직통 연락처(+선택적으로 다른 항공사 전체 공유 메일). 재고가 없으면 바로 전체 긴급 메일 |
| 5단계 (동일기종 타사) | 재고를 사전에 알 수 없으므로 **항상** 운영 항공사 전원에게 보내는 긴급 메일 |
| 6단계 Hand-carry | **통관팀** 연락처 + 실시간 항공편 API(더미)에서 조회한 가장 빠른 **예정편** 후보 |

긴급 메일은 **항상 정해진 템플릿으로 즉시 완성**되어 바로 발송 가능합니다. "✨ AI로 다듬기" 버튼을 누르면 로컬 LLM이 문장을 다듬어주며(선택 사항, 몇 초 소요), 실패해도 기본 문구가 그대로 유지됩니다. 각 조치는 처리 이력(Audit Log)에 자동으로 남습니다.

### 🗺️ 실시간 운항 현황
지도에 항공편 현재 위치(진행률 기반)와 공항별 정박 혼잡도를 표시하고, KPI 카드 + 정렬된 표로 한눈에 파악할 수 있습니다. **"🔄 새로고침"** 으로 즉시 갱신합니다. Hand-carry 편명은 여기서 조회하는 것과 같은 더미 API 공급자에서 나옵니다 — `FLIGHT_API_CONFIG`/`PARKING_API_CONFIG`를 켜고 표시된 지점에 실제 API 호출을 채우면 나머지 코드는 그대로 동작합니다.

### ⚙️ 데이터 관리
FAK/Allocation 재고, Pooling 계약, 공항별 Main Station 항공사, 기종별 운영 항공사, Allocation 부서·통관팀 연락처를 표로 직접 수정하고 **"💾 저장"** 을 누르면 `aog_db.json`에 즉시 반영됩니다. (Hand-carry 편명은 API 연동을 전제로 하므로 이 화면에서는 관리하지 않습니다.)

### 시나리오 1 — 1단계 FAK에서 즉시 해결

### 시나리오 2 — 2단계 Allocation까지 자동으로 확인 후 해결
> `B777-300ER 기종 ICN에서 AOG, 부품 BRK-B777-CARBON-01 필요`
- FAK 없음(자동 통과) → Allocation에서 확인 → 승인 시 Allocation 부서 연락처

### 시나리오 3 — 3단계 Pooling까지 자동으로 확인 후 해결
> `B737-800 기종 SIN에서 AOG, 부품 HYD-PUMP-737-11 필요`
- FAK·Allocation 자동 통과 → Pooling에서 SIA Engineering 확인 → 승인 시 파트너사 연락처

### 시나리오 4 — 4단계 Main Station 타사에서 해결
> `A321neo 기종 GMP에서 AOG, 부품 CAB-DOOR-ACT-321N-03 필요`
- FAK·Allocation·Pooling 자동 통과 → 4단계는 재고 여부와 무관하게 항상 멈춤 → 이번엔 제주항공 재고 확인됨 → 직통 연락 + (선택) 전체 공유 메일

### 시나리오 5 — 5단계는 항상 "직접 문의" → 즉시 작성되는 긴급 메일
> `B737-800 기종 ICN에서 AOG, 부품 WHL-NLG-737-22 필요`
- 1~3단계 자동 통과, 4단계에서 멈춤(재고 없음) → "다음 대안 찾기" → 5단계 도달, "사전에 알 수 없다"는 안내와 함께 B737-800 운영 항공사 전원에게 보내는 메일이 **즉시 템플릿으로 완성**되어 있습니다. 원하면 "✨ AI로 다듬기"도 눌러보세요.

### 시나리오 6 — 끝까지 가서 Hand-carry로 종료
> `A321neo 기종 CDG에서 AOG, 부품 FMS-A321-NEO-09 필요`
- 1~3단계 자동 통과 → 4단계 멈춤(재고 없음) → "다음 대안 찾기" → 5단계 멈춤(직접 문의 필요) → "다음 대안 찾기" → 6단계, 통관팀 연락처와 CDG행 실시간 예정편 후보가 뜹니다(시각은 실행 시점 기준으로 매번 새로 계산됩니다).

## 실 API 연동 방법 (참고)

`app.py` 상단부 "3.5. 실시간 항공편 / 공항 정박 데이터 공급자" 섹션에 있는 `FLIGHT_API_CONFIG` /
`PARKING_API_CONFIG` 딕셔너리의 `enabled`를 `True`로 바꾸고, `_fetch_raw_flight_feed()` /
