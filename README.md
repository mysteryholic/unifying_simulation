# unifying_simulation — AOG 자재 수급 어시스턴트

AOG(Aircraft on Ground) 상황 발생 시, 정해진 절차에 따라 자재를 수급하고 각 단계에서
실제로 연락/발송/조회까지 이어갈 수 있도록 돕는 Streamlit 대시보드입니다.
**모든 판단은 "데이터 관리" 화면에 저장된 데이터로만** 이루어지며, 5분 내 전 프로세스를
끝내야 하는 실제 AOG 상황을 전제로 속도와 정확성을 최우선으로 설계했습니다.
Google Colab 노트북 하나(`AOG_Dashboard.ipynb`)로 배포됩니다.

## 실행 방법

1. `AOG_Dashboard.ipynb`를 [Google Colab](https://colab.research.google.com/)에서 엽니다.
2. **Cell 1 → Cell 2 → Cell 3 순서대로** 실행합니다. (Cell 1은 강제 재시작 없이 설치만 합니다.)
3. Cell 3이 `google.colab.output`으로 앱을 새 창에 띄웁니다(제3자 터널 미사용, 사내망 안전).
4. 사이드바에서 **"🚨 AOG 대시보드 / 🗺️ 실시간 운항 현황 / ⚙️ 데이터 관리"** 를 전환합니다.

> ⚠️ 이전 버전에서 만든 `aog_db.json`이 남아 있어도 됩니다. 스키마 버전이 다르면 자동으로
> `aog_db.json.bak`으로 백업하고 새 스키마로 재생성하므로 `KeyError` 없이 바로 동작합니다.

## 이번 업데이트 내용 (v4 — 완전 데이터 기반 재작성)

이전 버전은 "저장된 데이터"와 "행동 알고리즘"이 따로 놀아서, 사실상 정해둔 시나리오 문구만
뱉는 문제가 있었습니다. 이번에 **데이터 모델과 알고리즘을 처음부터 다시 설계**해, 프롬프트에서
뽑아낸 값과 저장 데이터를 매칭해 프로세스가 실제로 데이터대로 움직이도록 고쳤습니다.

### 1. 입력은 '기번', 내부/타사문의는 '기종' (데이터로 매핑)
- 사용자는 **기번(Aircraft Registration, 예: `HL7702`)** 으로 입력합니다.
- `aircraft_registry` 데이터가 **기번 → 기종**을 매핑합니다 (`HL7702 → A330-300`).
- 타 항공사는 우리 기번을 모르므로, **Pooling·4·5단계 문의 메일은 '기종' 기준**으로 나갑니다.
- 등록되지 않은 기번을 입력하면 "등록부에 없습니다"라고 안내하고, 데이터 관리에서 추가하도록 유도합니다(멋대로 진행하지 않음).

### 2. 프롬프트에서 3요소 추출 — 기번 / 자재 파트넘버 / 공항
- 채팅에 "`HL7702 ICN에서 AOG, 부품 OXY-GEN-A330-15 필요`"라고 쓰면 세 값을 즉시(규칙 기반, 0초) 추출합니다.
- 한글 조사가 붙은 공항("ICN에서")도 정확히 인식합니다. 일부만 인식되면 무엇이 빠졌는지 알려주고 '직접 입력' 폼에 채워줍니다.

### 3. 같은 기번·자재라도 '발생 공항'에 따라 결과가 달라짐 (핵심 수정)
단계별 매칭 규칙을 데이터에 맞게 다시 정의했습니다.

| 단계 | 매칭 규칙 (데이터 기반) |
|---|---|
| 1 · FAK 키트 | 기체에 **싣고 다니는 키트**이므로, 기종별 `fak_kit` 리스트에 그 파트가 있으면 **발생 공항과 무관하게** 현장에서 사용 가능 |
| 2 · Allocation | `allocation_stock`의 `airport`가 **AOG 발생 공항과 일치**해야 사용 가능. 다른 공항에만 있으면 **바로 Pooling으로 넘어감** |
| 3 · Pooling | 파트너가 재고를 보유하고 **그 파트너의 `coverage_airports`에 발생 공항이 포함**돼야 지원 가능 (예: 싱가포르 파트너는 JFK를 커버하지 못함) |
| 4 · Main Station 타사 | **발생 공항을 Main Station으로 쓰는 항공사 ∩ 우리 기종 운영사**에게만 문의 |
| 5 · 동일 기종 타사 | **기종을 운영하는 모든 타 항공사**에게 문의 (타사 실재고는 기밀이라 항상 '문의' 단계) |
| 6 · Hand-carry | 통관팀 + 실시간 항공편 API에서 발생 공항행 가장 빠른 예정편 |

즉 **같은 `HL7702` + `IDG-A330-001`이라도** FRA에서 발생하면 Allocation(2단계)에서 끝나고,
ICN에서 발생하면 (거기엔 Allocation 없고 Pooling 파트너 커버리지도 안 맞아) 4단계까지 갑니다.
**같은 자재·기종이어도 공항에 따라 추천 경로가 완전히 달라집니다.**

### 4. 실시간 운항 현황 — 허브·주력 노선 시인성
"어느 공항이 허브이고 어떤 노선이 주력인지"가 한눈에 들어오도록 재구성했습니다.
- KPI 카드: 전체/운항 중 편수 + **최다 허브 공항** + **주력 노선**.
- 지도: 공항 마커 **크기 = 운항량**, **색 = 정박 혼잡도**(🟢🟡🔴), 노선 선 **굵기 = 운항 빈도**.
- **공항별 운항량(허브 순위)** / **주력 노선 Top** 랭킹 표(비중 막대 포함) + 상태·시간순 정렬된 항공편 표.

### 5. 스키마 버전 관리로 데이터 재연결 안정화
`aog_db.json`에 `_schema_version`을 넣어, 저장 파일 버전이 코드와 다르면 **자동 백업 후 재생성**합니다.
이전에 필드 추가 때마다 났던 `KeyError` 계열 문제를 구조적으로 차단했습니다.

## 데이터 관리 (판단의 유일한 근거)

"⚙️ 데이터 관리"에서 아래를 표로 직접 편집하고 저장하면 **즉시 프로세스 판단에 반영**됩니다.

- 🛩️ **기번 등록부** (`registration` → `aircraft_type`)
- 🧰 **FAK 키트** (기종별 키트 구성품 리스트: `aircraft_type`, `part_number`, `qty`)
- 📦 **Allocation 재고** (`airport` 포함 — 공항 일치 시에만 사용 가능)
- 🤝 **Pooling** (파트너 `coverage_airports` + 보유 재고)
- 🛫 **Main Station 항공사** / ✈️ **기종별 운영 항공사** (4·5단계 문의 대상 산정)
- 📇 **Allocation 부서·통관팀** 연락처

예: 기번 등록부에 `HL9001 → A330-300`을 추가하면, 다음 케이스부터 `HL9001`로 바로 프로세스를 시작할 수 있습니다. (검증 테스트로 확인됨)

## 테스트 시나리오

채팅창에 그대로 입력해 보세요. **같은 자재라도 공항이 다르면 결과가 달라지는 것**이 핵심입니다.

### A. FAK 키트에서 즉시 해결 (공항 무관)
> `HL7702 ICN에서 AOG, 부품 OXY-GEN-A330-15 필요`
> `HL7702 CDG에서 AOG, 부품 OXY-GEN-A330-15 필요`
- 둘 다 1단계에서 끝납니다 — 키트는 기체와 함께 이동하므로 ICN이든 CDG든 현장에 있습니다.

### B. Allocation은 공항에 따라 갈림 (같은 기번·자재)
> `HL7702 FRA에서 AOG, 부품 IDG-A330-001 필요` → **2단계(Allocation, FRA 거점)에서 해결**
> `HL7702 ICN에서 AOG, 부품 IDG-A330-001 필요` → ICN엔 Allocation 없음 → Pooling(LHT)도 FRA/CDG만 커버 → **4단계까지 진행**

### C. Pooling 커버리지 (같은 기번·자재)
> `HL8259 SIN에서 AOG, 부품 HYD-PUMP-737-11 필요` → **3단계(SIA Engineering, SIN 커버)에서 해결**
> `HL8259 JFK에서 AOG, 부품 HYD-PUMP-737-11 필요` → SIA는 JFK 미커버 → 3단계 실패 → **5단계 문의로 진행**

### D. 타사 문의는 기종 기준
> `HL8259 ICN에서 AOG, 부품 (희귀 파트넘버) 필요` → 5단계에서 B737-800 운영사(제주/티웨이/Ryanair/Qantas) 전원에게 **기종 기준** 긴급 메일. 이때 하단 "🛡️ 6단계 사전 준비"에서 통관팀 병행 요청도 가능.

### E. 등록되지 않은 기번
> `HL9999 ICN에서 AOG, 부품 OXY-GEN-A330-15 필요` → "기번 HL9999는 등록부에 없습니다" 안내(멋대로 진행하지 않음).

## 검증 내역

`streamlit.testing.v1.AppTest`로 다음을 자동 검증했습니다.

- 기번 → 기종 매핑, 미등록 기번 차단
- FAK 키트의 공항 무관 매칭, Allocation의 공항 일치 매칭, Pooling의 커버리지 매칭
- **같은 기번·자재라도 공항에 따라 종료 단계가 달라짐**(위 B·C 시나리오)
- 4단계 교집합(공항 기반 항공사 ∩ 기종 운영사), 문의 대상 없으면 5단계로 자동 통과
- 5단계 메일 본문이 기번이 아니라 **기종**으로 작성됨
- FAK 승인 후 정비사 `tel:` 링크·이력 기록, 4·5단계 Hand-carry 사전대비 블록
- **데이터 관리에서 기번/키트를 추가하면 다음 케이스가 그 데이터로 동작**(데이터 기반 확인)
- 구버전 스키마 `aog_db.json` 자동 백업·재생성(재연결 시 무중단)
- 실시간 운항 현황의 허브/주력 노선 랭킹·KPI 렌더링

## 실 API 연동 (참고)

`app.py`의 `FLIGHT_API_CONFIG` / `PARKING_API_CONFIG`를 `enabled: True`로 바꾸고
`_fetch_raw_flight_feed()` / `fetch_airport_parking_status()`의 표시 지점에 실제 API 호출을
채우면, 반환 스키마만 맞추면 지도·표·Hand-carry 후보 조회 등 나머지 코드는 그대로 동작합니다.
