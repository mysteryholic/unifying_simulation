# unifying_simulation — AOG 자재 수급 어시스턴트

AOG(Aircraft on Ground) 상황 발생 시, 정해진 절차에 따라 자재를 수급하고 각 단계에서
실제로 연락/발송/조회까지 이어갈 수 있도록 돕는 Streamlit 대시보드입니다.
Google Colab의 노트북 하나(`AOG_Dashboard.ipynb`)로 배포됩니다.

## 실행 방법

1. `AOG_Dashboard.ipynb`를 [Google Colab](https://colab.research.google.com/)에서 엽니다.
2. **Cell 1 → Cell 2 → Cell 3 순서대로** 실행합니다.
   - Cell 1은 설치 직후 런타임을 자동 재시작합니다. *"세션이 다운되었습니다"* 경고가 떠도 정상이니 무시하고 Cell 2부터 이어서 실행하세요.
   - Cell 3은 `google.colab.output`으로 앱을 새 창에 띄웁니다. npm/localtunnel 같은 제3자 터널을 전혀 쓰지 않으므로 사내망 보안 정책에 걸릴 일이 없습니다.
3. 새 창에서 대시보드가 열리면 사이드바에서 **"🚨 AOG 대시보드" / "🗺️ 실시간 운항 현황" / "⚙️ 데이터 관리"** 중 화면을 선택합니다.

## 화면 구성

### 🚨 AOG 대시보드 (메인 화면)
- 상단: **항공기 기종 / 자재 넘버 / AOG 발생 공항**을 드롭다운이 아닌 **직접 입력(텍스트)** 으로 받습니다. 대소문자·공백 차이는 자동으로 정규화되어 매칭됩니다.
- 좌측 절반 — **🤖 AI 상황 요약 & 추천 행동**: 케이스를 시작하거나 단계가 바뀔 때마다 **버튼 없이 자동으로** 브리핑이 표시됩니다. 기본은 로컬 LLM(Qwen2.5-0.5B-Instruct)이 "데이터 관리" 화면의 실데이터를 근거로 다듬은 문장이며, 모델이 없거나 응답에 실패하면 즉시 규칙 기반 요약으로 조용히 대체됩니다(앱이 멈추지 않음). 사이드바에서 "로컬 LLM 사용"을 끄면 항상 규칙 기반 요약만 즉시 표시됩니다.
- 우측 절반 — **📋 프로세스 진행**: 현재 단계 결과를 보여주고, **"▶ 다음 단계 진행"** 또는 **"✅ 현재 단계에서 조치(프로세스 종료)"** 를 사용자가 직접 눌러야 다음으로 넘어갑니다(자동 진행 없음 — 이 부분은 이번 업데이트에서도 그대로 수동 방식을 유지했습니다). 6단계(Hand-carry)는 최후 수단이라 "다음 단계" 버튼 자체가 없습니다.

### 단계별 종료 시 실제로 연결되는 조치
| 종료 단계 | 제공되는 조치 |
|---|---|
| 1단계 FAK | 케이스 시작 시 입력한 **요청 정비사 연락처**로 바로 걸 수 있는 `tel:` 링크 |
| 2단계 Allocation | 해당 공항의 **Allocation 부서** 전화(`tel:`)·메일(`mailto:`) 링크 |
| 3단계 Pooling | 매칭된 **Pooling 파트너사** 전화·메일 링크 |
| 4단계 (Main Station 타사) | 재고가 확인되면 해당 항공사 직통 연락처 + (선택) 다른 Main Station 항공사 전체에도 긴급 메일 가능. 재고가 없으면 바로 전체 긴급 메일 작성 |
| 5단계 (동일기종 타사) | 재고를 사전에 알 수 없으므로 **항상** 운영 항공사 전원에게 보내는 긴급 메일 작성 |
| 6단계 Hand-carry | **통관팀** 연락처 + 실시간 항공편 API(더미)에서 조회한 AOG 공항행 **가장 빠른 예정편 후보**를 보여주고, 선택한 편으로 확정 기록 가능 |

긴급 메일은 로컬 LLM이 본문을 작성하며(실패 시 고정 템플릿으로 자동 대체), **메일 작성/발송 자체는 여전히 "메일 작성하기" 버튼을 눌러야 진행되는 수동 방식**입니다. 각 조치는 처리 이력(Audit Log)에 자동으로 남습니다.

### 🗺️ 실시간 운항 현황 (신규)
- 지도에 **항공편 현재 위치**(출발~도착 사이 진행률 기반, 예정/비행 중/도착 상태 구분)와 **공항별 정박(Stand) 혼잡도**(색상: 초록=여유, 노랑=보통, 빨강=혼잡)를 함께 표시합니다.
- 아래에는 실시간 항공편 목록과 공항별 정박 현황 표가 나옵니다. **"🔄 새로고침"** 으로 즉시 갱신할 수 있습니다.
- 지금은 더미 데이터지만, 대한항공 편명 API가 실제로 연동 가능하다는 전제로 설계되었습니다 — `app.py`의 `FLIGHT_API_CONFIG`/`PARKING_API_CONFIG`를 `enabled: True`로 바꾸고 표시된 지점에 실제 API 호출을 채우면, 나머지 코드(맵/표/Hand-carry 후보 조회)는 수정 없이 그대로 동작합니다.

### ⚙️ 데이터 관리
FAK/Allocation 재고, Pooling 계약, 공항별 Main Station 항공사, 기종별 운영 항공사,
Allocation 부서·통관팀 연락처를 표 형태로 직접 추가/수정/삭제하고 **"💾 저장"** 을 누르면
`aog_db.json`에 즉시 반영됩니다. 노트북을 껐다 켜도 마지막 저장 내용이 유지됩니다.
(Hand-carry 편명은 API 연동을 전제로 하므로 이 화면에서는 관리하지 않습니다 — 위 실시간 운항 현황 참고.)

## 테스트 시나리오

값을 그대로 입력해 각 단계에서 프로세스가 어떻게 종료되고 어떤 조치가 뜨는지 확인할 수 있습니다.
(항공기 기종 / 자재 넘버 / AOG 발생 공항 순, 대소문자·공백은 무관합니다)

### 시나리오 1 — 1단계 FAK에서 즉시 해결 → 정비사 재연락
- `A330-300` / `OXY-GEN-A330-15` / `ICN`
- "요청 정비사 연락처"에 아무 번호(예: `010-1234-5678`)를 입력해두면, 종료 후 그 번호로 바로 거는 `tel:` 링크가 뜹니다.

### 시나리오 2 — 2단계 Allocation까지 가야 해결 → Allocation 부서 연락
- `B777-300ER` / `BRK-B777-CARBON-01` / `ICN`
- FAK 없음 → 다음 단계 진행 → Allocation 확인 → 종료 시 ICN 지점 Allocation 부서 연락처가 뜹니다.

### 시나리오 3 — 3단계 Pooling 파트너사까지 가야 해결 → 파트너사 연락
- `B737-800` / `HYD-PUMP-737-11` / `SIN`
- FAK·Allocation 없음 → Pooling에서 **SIA Engineering** 확인 → 종료 시 해당 파트너사 연락처가 뜹니다.

### 시나리오 4 — 4단계 Main Station 타사에서 해결 → 직통 연락 + (선택) 전체 공유 메일
- `A321neo` / `CAB-DOOR-ACT-321N-03` / `GMP`
- FAK·Allocation·Pooling 없음 → 4단계에서 GMP Main Station 항공사(제주항공) 확인 → 종료 시 제주항공 직통 연락처가 뜨고, 체크박스로 다른 Main Station 항공사 전체에도 공유 메일을 작성할 수 있습니다.

### 시나리오 5 — 5단계는 항상 "직접 문의" → AI가 작성한 긴급 메일
- `B737-800` / `WHL-NLG-737-22` / `ICN`
- FAK·Allocation·Pooling·4단계 모두 없음 → 5단계 도달 시 "재고 보유 여부는 사전에 알 수 없다"는 안내와 함께, B737-800 운영 항공사(제주항공/티웨이항공/Ryanair) 전원에게 보내는 긴급 메일이 자동으로 작성됩니다 — "현재 단계에서 조치"를 눌러 종료하면서 메일을 확인해보세요.

### 시나리오 6 — 끝까지 가서 Hand-carry로 종료 → 통관팀 + 실시간 편명 조회
- `A321neo` / `FMS-A321-NEO-09` / `CDG`
- 1~5단계 모두 없음("다음 단계 진행"을 5번) → 6단계에서 통관팀 연락처와 CDG행 **실시간 항공편 API(더미)** 가 조회한 가장 빠른 예정편 후보가 뜹니다. 편명·도착 예정 시각은 실행할 때마다 "현재 시각" 기준으로 다시 계산되므로 매번 조금씩 달라질 수 있습니다 — 원하는 편을 선택하고 확정할 수 있습니다.

### 입력 오류 / 수동 조기 종료 테스트
- 값을 하나라도 비운 채 시작하면 경고만 표시되고 프로세스는 시작되지 않습니다.
- 1~3단계에서 재고가 없는데도 "현재 단계에서 조치"를 누르면 별도 조치 버튼 없이 "수동 종료" 안내만 표시되며 정상적으로 이력에 남습니다.

### 데이터 관리 화면 테스트
- "⚙️ 데이터 관리"로 이동해 아무 표(예: FAK 재고)에 행을 추가하고 "💾 저장"을 누른 뒤, "🚨 AOG 대시보드"로 돌아와 방금 추가한 기종/자재 넘버로 케이스를 시작하면 1단계에서 바로 확인되는 것으로 변경 사항이 즉시 반영됨을 확인할 수 있습니다.

### 실시간 운항 현황 화면 테스트
- "🗺️ 실시간 운항 현황"으로 이동해 지도에서 항공편(삼각형 마커, 비행 중인 편만 표시)과 공항 마커(색상=정박 혼잡도)를 확인합니다. "🔄 새로고침"을 누르면 정박 현황 수치가 갱신됩니다(1분 단위로 값이 바뀌도록 설계되어 있어 같은 분 안에서는 동일할 수 있습니다).

## 실 API 연동 방법 (참고)

`app.py` 상단부 "3.5. 실시간 항공편 / 공항 정박 데이터 공급자" 섹션에 있는 `FLIGHT_API_CONFIG` /
`PARKING_API_CONFIG` 딕셔너리의 `enabled`를 `True`로 바꾸고, `_fetch_raw_flight_feed()` /
`fetch_airport_parking_status()` 안에 주석으로 표시된 지점에 실제 `requests.get(...)` 호출과
응답 파싱을 구현하면 됩니다. 반환 스키마(딕셔너리 키)만 기존 더미 데이터와 동일하게 맞추면
지도/표/Hand-carry 후보 조회 등 나머지 코드는 전혀 수정할 필요가 없습니다.
